Forms

Forms are used to handle user input.

Example

The best way to handle forms is using the Form plugin. This plugin takes care of input validation.

Auth token

As a security measure to prevent cross-site request forgery it is required to include an authentication token in forms. If the auth-token field is missing the form will not be processed and an error message is displayed.

Include the following line when creating forms that are send as POST:

Confirmation

Information send as GET does not require an authentication token to verify its integrity and can therefore not be trusted. When using a link to perform administrative tasks (e.g. ?id=1&action=delete to delete a page) confirmation should be requested from the user using $app->confirm(). The request is then resubmitted using POST.

Example
form->validate(array(
	));

if ( isset($app->GET_raw['action']) && $app->GET_raw['action'] == 'delete' )
{
	if ( !$app->POST_valid['confirm'] )
	{
		$app->confirm('Are you sure you wish to delete this page?');
	}
	else
	{
		// Delete the page
	}
}
?>

The $app->confirm() function terminates the code and displays a form to the user with an option to either proceed or cancel. If the user chooses to proceed the page is reloaded and $app->POST_valid['confirm'] is set to TRUE (this requires the Form plugin).

Handling user input

For security reasons $_GET and $_POST can not be used. It is better to use the filtered alternatives below.

Safe for HTML
  • $app->GET_html_safe
  • $app->POST_html_safe
Safe for database insertion (requires a database plugin)
  • $app->GET_db_safe
  • $app->POST_db_safe
Unfiltered (and unsafe!)
  • $app->GET_raw
  • $app->POST_raw
